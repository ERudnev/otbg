## Position: `uint32_t` для координат

**Где:** `src/Game/Position.hpp`

**Суть:** координаты и конструктор на беззнаковых `uint32_t`. Для клеток на карте это кажется естественным, но неудобно для любой «векторной» арифметики.

**Почему цепляет:**
- вычитание легко уезжает «левее нуля» (underflow);
- направление / шаг / delta (отрицательная скорость) нормально не выразить;
- в `Map.cpp` уже костыли: `int64_t` для дистанции, а в `tryGetNextPosition` — `nextPosition.x -= 1` по `uint32_t` (работает только потому что декремент делается под guard'ом `target < current`).

**Возможные направления (не решено):**
- `int32_t` для координат и delta;
- отдельный тип `Vec2` / `Delta` (signed) vs `Cell` / `Position` (unsigned, только валидные клетки);
- оставить как есть до рефактора `Core`, если карта останется единственным потребителем.

**Когда трогать:** при переписывании `Map`/движения или если появятся механики с относительным смещением (ворон −2 клетки, стены, зоны).

## Юнит (и карта?) — разложить на функциональности

**Где:** `src/Game/Unit.hpp`, `World::step()`, `Map` — но скорее про будущий `Core`/`Features`

**Суть:** при чтении требований к архитектуре (охотник, яд, дальняя/ближняя атака, башня 2×2, летающий ворон, огненная стена…) монолитный `Unit` с полями мечника выглядит тупиком. Хочется резать сущность на части по **ролям/способностям**, а не по типам юнитов одним классом.

**Почему цепляет:**
- `Unit` уже смешивает: identity, позицию, march, HP, статы, способность мечника (`chance`, `rending`), бой;
- `World::step()` знает конкретную логику мечника (соседи → ability → attack → move);
- охотник потребует другие статы (`Range`, `Agility`, `Poison`), другие правила выбора цели и DoT;
- башня / ворон / маг — ещё другие «куски» (не двигается, не занимает клетку, AoE-эффект на карте);
- карта тоже может перестать быть «просто сеткой юнитов»: стены, зоны урона, footprint 2×2.

**Пока не ясно — как именно:**
- **компоненты** (data-only: `Health`, `Movable`, `RangedAttack`, `PoisonOnHit`) + система, которая их обходит;
- **роли / capabilities** (интерфейсы: `CanMove`, `CanAct`, `OccupiesCell`) — юнит = набор ролей;
- **aspects** / **effects** (яд, рваные раны, огненная стена как состояние на сущности или клетке);
- **features-модули** поверх тонкого `Core` (скорее всего ближе к ТЗ, чем ECS «ради ECS»).

**Заметка себе:** не обязательно выбирать терминологию сейчас. Важнее **ось декомпозиции**: «что юнит *умеет*» и «что на нём *висит*» — отдельно от «это мечник».

**Возможные направления (не решено):**
- минимум: вынести `TurnAction` / `Behavior` per unit type, `Unit` = id + hp + position + указатель на behavior;
- средне: effect list на юните + stat block; карта хранит cell modifiers;
- тяжело: полноценный ECS — скорее overkill за 2 дня, но directionally близко к «20 джунов добавляют юнитов».

**Когда трогать:** перед реализацией охотника и переносом в `Core`/`Features` — иначе каждый новый юнит раздует `Unit` и `World::step()`.

## Цикл Карта ↔ Юнит — занятость через множество `Unit`

**Область:** декомпозиция игры.

**Где вылезло:** baseline `src/Game/Map.hpp` — `_positionByUnitId`; при переносе в `NewGame` — `entity::Unit` с `Map::Position` и `entity::Map` как отдельные сущности.

**Проблема:** карта знает о юнитах (индекс `unitId → position`), юниты знают о карте (тип `Map::Position`, правила сетки). Получается цикл зависимостей и смешение ролей: «геометрия поля» и «кто где стоит» живут в одном месте.

**Рассматривалось:** вынести занятость в отдельный `Body` / `Occupancy` — физическое тело с позицией и HP, множество которых заменяет `Map::_positionByUnitId`.

**Что решено сейчас (❓ — рабочий компромисс, не финальная модель):**
- **`Unit` остаётся `Unit`** — не дробить ради разрыва цикла. Важно сохранить логику «кто ходит, кого бьют, а кто тут просто дерево»; отдельный `Body` размывает эту грань.
- **Юниты по умолчанию занимают свою клетку** — позиция (и HP) остаются у `Unit`, как сейчас.
- **Множество всех `Unit` отвечает на «что тут занято, а что — нет?»** — вместо того чтобы карта хранила `_positionByUnitId`, занятость выводится из юнитов (или из индекса поверх них). `Map` — только размер и геометрия сетки.

**Почему ❓, а не ✅:** цикл `Map ↔ Unit` формально ещё есть (юнит всё ещё тащит `Map::Position`), но сознательно принимаем это ради простой семантики `Unit`. Полный разрыв через отдельную сущность занятости отложен.

**Когда появятся птички — переиграть:** если «occupance by default» перестанет работать (летит → клетка под ним свободна / занята / атакуема по другим правилам), вернуться к вопросу определения `Unit`. Тогда либо снова выделить `Body` / `Occupancy` как spatial-слой, либо явно пометить часть юнитов как «не занимают клетку по умолчанию» через effect / capability — см. также заметку про птичек. Сейчас не строим модель под гибридный полёт заранее.

## Кеши как отдельная ось `category::Entity` — «данные типа»

**Где всплыло:** `sw::game::Map::_positionByUnitId` в baseline — кеш «какие клетки заняты юнитами»; в `NewGame` без явного места под такие вещи приходится каждый раз пересчитывать.

**Идея:** у `category::Entity` (сейчас `State` / `Logic` / `Emitters`) добавить ещё одну ось — **данные типа**: производные / кешируемые представления, которые не первичное состояние сущности, а удобный срез для запросов. Туда же ложились бы индексы вроде `unitId → position`, spatial lookup «свободна ли клетка», и прочие «можно вывести из мира, но дорого каждый раз».

**Почему не делаю сейчас:** выглядит полезно и архитектурно честно, но на это нет ни времени, ни вдохновения. Пока за отсутствием оси платим вычислениями там, где в прототипе уже лежал кеш.

## Conjurer — юнит, который создаёт юнитов

**Идея:** гипотетический `Conjurer` — сильный стресс-тест архитектуры. Не «ещё один тип с другими статами», а юнит, который **порождает других юнитов** и **шлёт миру команды** (spawn, march, …). Текущая модель «сценарий парсится один раз → юниты только ходят» это не покрывает. Повод подумать: кто владеет миром, кто может мутировать его mid-simulation, как не превратить `World` в god object.

## Детерминизм baseline перед рефакторингом

**Где вылезло:** на стадии фиксации до-рефакторенного `src/Game` сценарными тестами.

**Проблема:** в `Unit.cpp` использовался один process-wide источник случайности. Для одиночного запуска это допустимо, но в `scenario_runner`, где несколько сценариев выполняются последовательно в одном процессе, baseline становился недетерминированным.

**Подход:** так как `src/Game` рассматривается как baseline/reference implementation, глубокий рефакторинг этого слоя на данном этапе был нежелателен. Вместо протаскивания RNG через сигнатуры и вместо введения новых runtime-абстракций в baseline добавлен локальный hook для инициализации источника случайности снаружи. `Main` инициализирует его один раз для обычного запуска, `Test` — сбрасывает фиксированным seed перед каждым сценарием.

**Результат:** baseline стал воспроизводимым без изменения общей формы старой реализации. Это даёт воспроизводимые сценарные tests, дебаг, реплеи и детерминированную симуляцию в инструментах и CI, не подменяя собой целевой архитектурный рефакторинг.

## `TheGame/Core` и `TheGame/Features` как разные библиотеки

**Идея:** `TheGame/Core` и `TheGame/Features` должны со временем разъехаться в две отдельные библиотеки с односторонней зависимостью `Features -> Core`.

**Цель:** жёстко запретить `Core` знать о конкретных фичах и типах контента. `Core` должен задавать базовые сущности, контракты и механизм построения/подключения, а конкретные игровые возможности должны приезжать сверху, из `Features`.

**Почему это важно:**
- исчезает соблазн тащить в `Core` знания о мечнике, охотнике, яде, башнях и прочих конкретных механиках;
- архитектурная граница становится не «по договорённости», а проверяемой на уровне линковки и зависимостей;
- проще наращивать набор фич, не размывая ядро;
- легче обсуждать extension points: что именно `Core` обязан предоставить `Features` для сборки игры.

**Что должен дать `Core`:**
- базовые типы и state-модель;
- world/update loop;
- точки расширения для регистрации сущностей, поведения, эффектов, команд и/или фабрик;
- механизм композиции, чтобы игра собиралась из набора фич, а не из знания `Core` о них.

**Когда трогать:** перед серьёзным ростом `NewGame`, когда появится 2+ независимых feature-модуля и начнётся реальное давление на границу между «движком» и «контентом».

## 4-осевое разделение фичи на `State/Operations/Rules/Emitters`

**Идея:** разложить каждую feature по четырём осям: `State`, `Operations`, `Rules`, `Emitters`. Это даёт очень аккуратную мыслительную рамку: отдельно хранить состояние, отдельно описывать действия, отдельно валидировать/ограничивать их правилами и отдельно эмитить события/побочные эффекты.

**Почему пока под вопросом:** модель красивая, но есть риск, что для текущего масштаба она создаст слишком много отдельных коробочек и заставит всё время прыгать между файлами. Как архитектурный ориентир идея сильная, но пока не очевидно, что цена разделения оправдана.

---

## 3-осевое разделение фичи: `State/Logic/Emitters`

**Идея:** более приземлённая альтернатива — схлопнуть `Operations` и `Rules` в одну ось `Logic`, оставив три части: `State`, `Logic`, `Emitters`.

**Почему нравится:** есть сильное ощущение, что именно в таком виде система получится проще и честнее:
- `State` хранит данные;
- `Logic` делает всё содержательное: проверяет правила, принимает решения и применяет изменения;
- `Emitters` отвечают за внешние события и наблюдаемость.

**Почему это выглядит как рабочий компромисс:** сохраняется полезное разделение ответственности, но без лишней дробности. Для текущего проекта это, похоже, даёт хороший баланс между архитектурной чистотой и простотой разработки.

## Контейнеры стёртых типов как способ скрыть знание о мире

**Где всплыло:** `src/NewGame/Core/State/Structure/Composition.h`

**Идея:** контейнеры стёртых типов здесь интересны не сами по себе, а как один из способов стереть информацию о мире и давать объектам мир только в виде контекста.

**Смысл границы:** если код не знает типа фичи, он не может:
- инстанцировать шаблонный getter;
- ни к чему привести соответствующий стёртый слой;
- вообще осмысленно поговорить об этом типе.

**Практический эффект:** без include конкретного типа для тебя этого слоя как будто не существует. Контейнер знает, что он есть, но наружу не протаскивает знание о нём автоматически.

**Почему это всё же хочется тащить в архитектуру:** альтернатива выглядит хуже — `Context` как `GodObject`, который знает вообще всё и всем это знание раздаёт. Наш "стёртый" контейнер тоже в каком-то смысле знает всё, но делает это намного более тихо: без знания конкретного типа он ничего о нём никому не скажет.

**Почему пока под вопросом:** идея кажется архитектурно полезной, но всё ещё требует аккуратной проверки на сложность, читаемость и цену сопровождения.

## `Failure Events` как реакция на невалидные команды мира

**Где всплыло:** старый `createMap()` и вопрос "что делать при втором вызове?" в `src/Game/World.cpp`.

**Идея:** если в мир приходит команда, которую в текущем состоянии нельзя корректно применить, не падать и не молчать, а эмитить специальное событие ошибки вроде `MapCreateRejected`, `InvalidCommand`, `WorldFailure` и т.п.

**Почему не хочу идти туда сейчас:** это выглядит как минимально "приличная" реакция архитектора, но быстро начинает размывать модель доменных событий. Вместо событий о том, *что произошло в мире*, появляются события о том, *почему мир отказался что-то сделать*, и граница между доменной логикой, валидацией и диагностикой начинает плыть.

**Что в этом не нравится:**
- неудача команды начинает жить как будто это такой же first-class domain fact, как движение, атака или смерть;
- протокол событий разрастается диагностическими сущностями;
- появляется соблазн лечить архитектурные неясности новыми `...Failed` / `...Rejected` вместо того, чтобы жёстче определить контракт команды и допустимые состояния мира.

**Предпочтение вместо этого:** лучше явно решить контракт на уровне `Logic`/API мира: команда допустима или нет, второй `createMap()` разрешён или запрещён, и что именно должен сделать вызывающий код. Диагностику при необходимости лучше держать отдельно от основной доменной ленты событий.

## Типы живут у своих фич, наружу торчат через export-facade

**Где всплыло:** при выносе примитивов из `Features/Types.h` в их смысловых владельцев и появлении `src/NewGame/Features/_types.exports.h`.

**Идея:** доменные типы не должны жить в общей свалке, если у них уже есть очевидный хозяин. `Map::Position`, `Body::HitPoints`, `MeleeAttack::Strength`, `RendingAbility::Chance` и подобные вещи лучше держать рядом с той feature, которой они реально принадлежат. Наружу их можно временно экспортировать короткими alias'ами через отдельный facade-файл, чтобы не ломать весь внешний слой сразу.

**Почему нравится:**
- код перестаёт врать о происхождении типов;
- зависимости между кусками модели становятся честнее;
- меньше ложной "общности" у вещей, которые на деле принадлежат разным механикам;
- проще расти дальше, не превращая `Types.h` в бесконечную свалку.

**Цена:**
- растёт число include-зависимостей между feature-хедерами;
- сигнатуры наружного API перестают быть "безликими интами" и начинают требовать понимания, что именно туда передаётся;
- поднимается реальная проблема интерфейса наружу: раньше можно было пихать числа и не париться, а теперь нужно осознанно решить, какие типы экспортировать наружу как facade, а какие оставлять внутренними;
- местами появляются спорные вопросы владения типом (`Position`, `Chance`, `Poison`) и их всё равно приходится решать архитектурно, а не прятать в общую помойку.

**Почему всё равно принимаю:** это выглядит как здоровое ужесточение модели. Цена есть, но она платится за честность границ, а не за декоративную сложность. Назад к общей свалке типов откатываться уже вряд ли захочется.

## Порядок апдейта выводить из зависимостей типов

**Идея:** выразить порядок `step()` / update между сущностями, ability и effect не вручную, а через зависимости между типами. Например, чтобы граф типов сам подсказывал, в каком порядке обновлять `Unit`, `Ability`, `Effect`, и мир мог бы проходить по топологически выведенным фазам.

**Почему не делаю сейчас:** идея красивая, но для текущего тестового это уже следующий этаж сложности. Вместо того чтобы шлифовать боевую модель, пришлось бы проектировать отдельный механизм orchestration поверх type-graph и долго спорить, где именно заканчиваются зависимости данных и начинаются зависимости апдейта.

**Что выбираю вместо этого:** тупо и честно хардкодить порядок в логике мира и в его `step()`. Это осознанный компромисс: `World` знает, кого и в каком порядке дёргать, а если система доживёт до следующего витка сложности, тогда уже можно будет вытащить update-order в более общий механизм.

**Почему это ок:** для тестового задания важнее иметь ясный и управляемый orchestration, чем строить красивый, но дорогой мета-рантайм. Формула простая: доживём - сделаем.

## Птичек моделировать через 3D-позицию или отдельный эффект полёта

**Идея:** если в игре появятся птички, которые умеют то летать, то приземляться, то `entity::Unit::position` рано или поздно упрётся в новую развилку. Один путь — расширять позицию до трёхмерной модели. Другой — оставлять карту двумерной, а состояние "сейчас летит" выражать отдельным эффектом / состоянием, которое меняет правила движения, занимаемости клетки и доступности для атак.

**Почему не делаю сейчас:** это уже не столько вопрос архитектуры движка, сколько вопрос игрового дизайна. Что именно означает "летит" в конкретной игре: находится ли юнит над клеткой, можно ли под ним стоять, когда именно он считается приземлившимся, как это влияет на melee/range/occupancy — всё это зависит от дизайнерского решения, а не от желания заранее построить универсальную модель.

**Что выбираю вместо этого:** пока оставить `Unit::position` как позицию на карте и не пытаться заранее угадать язык для гибридного полёта. Если такая механика реально придёт, тогда уже честно решать: нужна ли третья координата, отдельный effect, или комбинация spatial-state и правил на атаки/занимаемость. **Триггер для пересмотра:** заметка «Цикл Карта ↔ Юнит» — там «occupance by default» может перестать хватать, и придётся переиграть, что такое `Unit` vs отдельная занятость клетки.

## Внешние ID юнитов + маппинг вместо внутренних идентификаторов модели

**Где вылезло:** входной контракт — `SPAWN_SWORDSMAN`, `SPAWN_HUNTER`, `MARCH` и т.п. задают `unitId` снаружи; baseline и сценарии уже завязаны на это.

**Проблема:** по-хорошему модель должна сама выдавать свои внутренние идентификаторы (`createEntity` / runtime id), а не принимать их из API. Но **входной контракт неприкосновенен** — снаружи по-прежнему приходит «юнит №1», «юнит №2». Значит, пришлось бы заводить **внешние ID** и держать в мире **маппинг** `externalId → internalId`.

**Зачем ещё это могло бы понадобиться:** защита от неявной привязки «ходим юнитами в порядке добавления». Порядок добавления **вообще не обязан** совпадать с внешними ID — в требованиях это описано неоднозначно, и легко начать опираться на порядок в контейнере там, где контракт говорит только про `unitId` из сценария.

**Почему ❌ — вряд ли буду делать:** цена высокая (слой маппинга, два пространства id, дублирование в `World`), а выигрыш для текущего масштаба сомнительный. Проще жить с тем, что **внешний `unitId` из сценария = внутренний `Unit::Id`**, и явно **не** выводить turn-order из порядка вставки в `unordered_map` / порядка spawn в файле — только из осознанного списка/очереди в `World`, если он понадобится.

**Что выбираю вместо этого:** один id end-to-end как в baseline; порядок хода — отдельное явное решение в логике мира, не побочный эффект storage order.
